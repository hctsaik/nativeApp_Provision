發佈 GUI — 照著圖做版本發佈（實測截圖版）

2026-07-19 由腳本驅動真實 GUI 對真實 nativeApp 跑完整旅程產生， 並真的把產出的 release 安裝到一台「目標機」目錄：每張圖都是實截畫面、每段對話框與 安裝輸出都是逐字引用（工作區路徑縮寫為 <WS>）。原理與 CLI 版見 docs/release-e2e-step-by-step.html。
你不需要管金鑰：發行金鑰第一次發版時自動建立、發行者信任清單自動隨 release 帶到目標機並在首次安裝時釘住——全程不出現在你的操作裡。

CIM 平台交付指示（production-1.0.0）
1. 把整個資料夾複製到目標機（USB 可；不要經 OneDrive 同步夾）：
   <WS>\gui-workspace\releases\production-1.0.0
2. 目標機雙擊資料夾裡的 install.bat——首次＝安裝，之後拿新的
   release 資料夾再雙擊同一顆＝更新（使用者資料不會動）。
3. 啟動：安裝根目錄的 bin\start-platform.bat（預設 %LOCALAPPDATA%\CIM-Platform）。
注意：發佈機的金鑰目錄（<WS>\gui-workspace\keys）絕不複製出去。

目標機｜安裝與更新（對方要做的全部）

release 資料夾自帶安裝器。對方拿到資料夾後只有兩個動作：

  1. 雙擊資料夾裡的 install.bat——首次＝安裝（自動釘住發行者）， 之後拿到新的 release 資料夾、再雙擊裡面同一顆＝更新（使用者資料完全不動）。
  2. 啟動：安裝根目錄的 bin\start-platform.bat （預設裝在 %LOCALAPPDATA%\CIM-Platform）。

本次實測（把上面 GUI 產的 production release 真的裝到一台目標目錄）的完整輸出：

> rem 目標機:雙擊 release 資料夾裡的 install.bat(以下為實際輸出)
[install] 首次安裝：發行者已釘住（<WS>\target-machine\trusted_publishers.json）
[install] 結果:UPDATED  啟用版本:1.0.0
[install] 完成。啟動:<WS>\target-machine\bin\start-platform.bat
[install] 之後更新:拿新的 release 資料夾,雙擊裡面的 install.bat 即可(資料不動)。

安全性在背後默默工作：首次安裝把發行者清單釘住在安裝根目錄，之後每次更新 都用釘住的那份驗章——就算有人把更新資料夾連簽章帶清單整個掉包，也會被拒絕、 裝置維持原版本（有自動化測試證明）。

出狀況怎麼辦

畫面意義與下一步
「先修這些再發版」清單每一條都寫了怎麼修（例：版本號已存在→換欄位旁建議值；缺殼→build-shell.bat）。修完直接再按一次。
按了「取消」整棵程序樹被終止；「本次才出現」的半成品自動清掉，對話框據實列出清了什麼——同一版本可直接重跑。
失敗對話框輸出區的 [FAIL] 行就是原因；半成品同樣已清，修正後重跑同版本。
目標機 install.bat 顯示 FAILED/untrusted更新包驗不過釘住的發行者——包被動過或發行金鑰換了。重新從發佈機拿正式 release；真的換了金鑰才需要管理員更新目標機的信任清單。

金鑰去哪了？（你不用管，但可以知道）

私鑰在發佈機的 %USERPROFILE%\.cim-keys（工作區之外，永遠不會跟 releases 一起被複製上 USB）；公鑰清單自動放進每一份 release、目標機首次安裝時釘住。 要換金鑰（外洩/輪替）才需要動它——那是管理員操作，走 CLI（release.py keygen）。

產生方式：scratch workspace 內以 Tk 事件驅動 release_gui.py、PIL 實截視窗、 對話框攔截逐字記錄；目標機安裝為真實 subprocess 執行 release 內建 device_install.py。 後端契約見 tests/test_release_gui_backend.py 與 tests/test_device_install.py。
