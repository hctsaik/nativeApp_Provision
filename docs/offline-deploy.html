讓模組在沒有網路的電腦上跑起來

工具的重相依（torch 那種）不裝在平台核心，而是宣告在各工具的 plugin.yaml，由平台在工具首次啟動時裝進專屬的 venv。 離線機沒有 PyPI，所以那些 wheel 必須事先下載好帶過去。 這份文件講怎麼帶、怎麼裝，以及在畫面上按下 Start 之後到底會看到什麼。

Part 1 · 在連網機上準備補給包

先看清單，不下載

--dry-run 只掃描不抓檔。要看兩件事：工具數對不對，以及 目標標籤那一行。

這是最容易出事的地方

曾經有人在 Python 3.14 的機器上 pip download，抓到 51 個 cp314 的 wheel，帶到鎖 3.11 的平台上一個都裝不起來。 本工具永遠把 --platform / --python-version / --abi 明示傳給 pip，就是為了讓那件事不可能再發生。

產包

下載時間視網路而定。它替你做了兩件不會想手動做的事：把單檔超過門檻的 wheel 搬進 big-deps\（跨工具只存一份），以及產完立刻用 pip download --no-index 對本地 wheelhouse 重解一次依賴圖—— 在開發機就證明這包離線裝得起來。

讀 REPORT.md，決定怎麼搬

第一個實質區塊就是大型相依——因為它決定你要不要分開搬運。

Part 2 · 在沒有網路的電腦上

複製後先驗一次

搬運會損毀檔案。verify 逐檔重算 sha256，抓得到改了一個 byte 的 wheel、被截斷的檔案、缺檔、以及清單上沒有的多餘檔案。它不需要平台專案， 隨身碟插上就能驗。

找出 --deppack-cache 要指到哪裡

這是唯一需要動腦的參數。它必須等於平台 engine 的 CIM_DEPPACK_CACHE。

啟動方式--deppack-cache 指到
可攜模式（start.bat）<APP_ROOT>\data\<project-key>\deppack-cache
開發模式（start-dev.bat）<平台專案>\sidecar\python-engine\.deppack-cache
有自訂 CIM_DEPPACK_CACHE該環境變數的值

可攜模式的 <project-key> 是 start.bat 算出來的 （<專案資料夾名>-<絕對路徑sha256前8碼>）。 先雙擊一次 start.bat 讓 data\<project-key>\ 長出來，再照抄那個路徑——順便確認 app 本身跑得起來。

套用

apply 做的事很少，這是刻意的：

  • 它只搬檔案：把 packs\<工具>\ 放到目標位置，把 big-deps\ 的大 wheel 補回各工具的 wheels\。
  • 它不執行 pip、不連網。真正的安裝是平台 engine 在工具首次啟動時做的。
  • 它採「暫存組裝 → 全量 sha256 驗證 → 原子性換位」。中途失敗或斷電時目標維持原樣， 絕不留下 wheels 不完整但 deppack.json 存在的目錄。
如果 big-deps\ 分開搬運、還沒放回去

暖機：先把相依裝好，再開 GUI

別跳過這一步

Tauri 殼的 HTTP bridge（bridge.rs::api_post）對 engine 的請求有 30 秒逾時；而 engine 是在 POST /tools/<id>/start 的處理過程中同步安裝相依的。torch 級的相依實測要 76 秒—— 殼會先放棄，畫面顯示「Failed to start tool: undefined」。 （相依其實裝完了，只是沒人收到回應。）

warmup.py 把安裝成本移出「按下 Start」那一刻。它跟 apply 的分工是： apply 搬檔案，warmup 真的裝。warmup 借用平台自己的 core.tool_deps——同一套驗章、同一句 pip --no-index—— 所以它需要平台專案，而且必須用平台會用的那顆 Python 跑 （venv 的 ABI 綁在建立它的直譯器上）。

啟動平台，在畫面上把工具點開

照平常方式啟動（start.bat 或 start-dev.bat）。 下面每一張都是 Playwright 經 CDP 連上真實 WebView2 拍的， 而且整台機器的 pip 索引被指向死位址——任何連網嘗試都會立刻失敗。

這份說明書是被證明的，不是被宣稱的

上面的每一步都跑過一次自動化 E2E：真的 Tauri 殼、真的 WebView2、真的 Streamlit 子程序。三個對照組，每組都把 PIP_INDEX_URL 指向 http://127.0.0.1:1/simple（死位址）。 若 engine 的安裝路徑漏掉 --no-index，pip 會去連那個位址並失敗，B 組就會紅。

判準刻意用三個互相獨立的證據，缺一不可：iframe 畫出真正的 UI （而不是 Python traceback）、engine.log 說相依 ready、以及直接拿該工具 venv 的 python.exe 去 import torch。

A · 忘了套用補給包

平台的 _prewarm_deps_and_timeout() 註解寫著 never block launch on dep handling——相依裝不起來時，engine 只記一行 warning，照樣把工具啟動起來。所以使用者不會看到「工具打不開」， 而是看到工具打開了、畫面上是一段 ModuleNotFoundError。 這只有把 GUI 真的點開才會發現；CLI 測不到。

B · 套用了，但跳過 warmup 直接按 Start

engine 開始在請求裡裝 torch。33 秒後， 殼的 HTTP bridge 逾時，portal 顯示「Failed to start tool: undefined」—— 但 engine 沒有停，它在背景把相依裝完、也把 Streamlit 拉起來了。

解法是再按一次 Start：這次相依已在 venv 裡，指紋命中， engine 幾秒內就回應。所以 B 最終仍然算繪成功（總計 82 秒）—— 只是中間嚇了使用者一跳。第 7 步的 warmup.py 就是為了讓這一跳不會發生。

順帶修掉一個假陽性

平台既有的 E2E harness 只檢查 [data-testid="stApp"] 存在且沒有 “Not Found” 就判 RENDERED。但 Streamlit script 在 import 階段崩潰時， stApp 容器仍然存在——A 組那張錯誤畫面會被判成通過。 本次的 harness 因此改成必須「有內容且不是 traceback」，並額外要求 engine.log 與 venv 的兩項獨立證據。

建議給平台的修正

真正的根治是讓 bridge.rs 對 POST /tools/<id>/start 放寬逾時（或把相依安裝改成非同步、回報進度）。在殼重編之前， warmup.py 是不需要動 Rust 的完整解。 （本機 WDAC 擋 cargo，殼要在別台機器重編。）

日常維護

工具改了 requires: — 在連網機重跑同一條 build 指令。 它是增量的：requires 指紋、python 標籤、平台標籤、既有包的 sha256 驗證， 四項全對就沿用快取，不會重抓 195 MB 的 torch。想全部重來加 --force。

某個工具在離線機上壞了 — 刪掉它的 venv （data\<key>\tool-venvs\<tool_id>\），重啟平台， 下次點開時 engine 會從補給包離線重建。補給包不用重搬。

搬移整個 <APP_ROOT> 之後 — venv 內含絕對路徑。 搬完刪掉 data\*\tool-venvs\，重啟，離線重建。

為什麼設計成這樣

為什麼不把 torch 裝進平台核心，讓所有工具共用？

三個理由。版本綁架：不同工具對同一個套件的版本約束會打架， 同一個 site-packages 只能犧牲其中一個。部署矛盾：可攜模式的賣點是 「複製資料夾＝部署」，torch 進核心會讓每台機器、每個專案都背著 2 GB， 即使它根本不跑 AI 模組。卸載不乾淨：工具移除後， 它留在共用環境裡的套件沒人敢刪。

為什麼 apply.py 不直接跑 pip 裝好？

因為安裝需要知道「裝進哪個 venv、用哪顆 Python、什麼時候裝」，那是平台 engine 的知識。 apply.py 只負責把檔案放到 engine 找得到的位置——職責單一、失敗模式少、 可以在只有標準函式庫的環境跑。而且 engine 的驗章（fail-closed）因此仍然是最後一道關卡。

為什麼大相依要獨立一個資料夾，而不是壓縮進去就好？

因為「知道大東西在哪」本身就是使用者要的資訊。195 MB 的 torch 跟其餘 43 個 wheel 加起來是 342 MB，郵件寄不了、隨身碟要挑。把它單獨拉出來，使用者可以自己決定怎麼搬運。 deppack.json 保持完整、apply 時用 sha256 驗證重組， 所以「分開搬運」不犧牲任何正確性保證。
