CIM 平台版本發佈 — E2E 教學（實測版）

2026-07-19 於開發機實際跑完整條發佈鏈產生：所有指令輸出逐字擷取、 所有時間與大小皆實測、截圖為真實 WebView2 視窗（非示意圖）。 工作區絕對路徑以 <WS> 縮寫顯示，其餘未經修改。
本次實測對象：真實 nativeApp 專案（engine 2,815 個檔案、含 485MB vendor， .napp 成品 441MB）。

一句話心智模型：發佈 = 把平台打成一顆簽了名的 .napp， 放進一個機器可驗的 release 資料夾；User 機器 = 從 release 資料夾安裝進 「不可變版本槽」，重啟才切換，切壞了可回滾。你從頭到尾不會覆蓋任何正在使用的檔案。
發布機（連網/開發機） User 機器（可離線） ────────────────────────── ────────────────────────── release.py keygen ← 一次性 複製 release 資料夾（USB 可） release.py pack-platform ┐ release.py verify ← 可出貨嗎? release.py sign │ 每次發版 NativeAgent.update ← 驗章+裝進版本槽 release.py build │ platform_launcher ← 啟動（讀 active.json） release.py verify ┘ （關閉重開才換版；壞版自動回滾） release.py promote ← internal→production

0. 一次性準備：發行金鑰與信任清單

只做一次。產生 Ed25519 發行金鑰對；私鑰留在發布機（勿進 repo、勿進交付包）、 公鑰進 trust store（之後放到每台 User 機器上）。

> py -3.11 release.py keygen --key-id fab-team --out keys\fab-team.private.json --trust-store keys\trusted_publishers.json
OK: 私鑰已寫入 <WS>\keys\fab-team.private.json（保管於 secret store，勿進 repo/包/裝置）
OK: 公鑰已加入 trust store <WS>\keys\trusted_publishers.json
金鑰三條紀律： ① 私鑰檔（*.private.json）只放發布機的加密位置，遺失＝重發金鑰、外洩＝立即從 trust store 移除該 key_id； ② trust store（trusted_publishers.json）隨部署放到 User 機器 apps\<app>\； ③ 換鑰＝keygen 新 key_id 加入 trust store（舊鑰標 retired 仍可驗舊版，撤銷＝整條移除）。

1. 每次發版四步：pack → sign → build → verify

1-1. 把平台打成 .napp（實測 62 秒 / 2,815 檔）

engine 樹進 payload（__pycache__、.tool-venvs、logs、 tools.sqlite 等衍生物自動濾掉）；17MB 的 Tauri 殼以內容定址 blob 旅行，不佔 .napp。

> py -3.11 release.py pack-platform C:\code\claude\nativeApp --version 1.0.0 --out work\cim-platform-1.0.0.napp --blobs work\blobstore
OK: <WS>\work\cim-platform-1.0.0.napp（2815 個檔案，殼 blob 5d250bed13cf2668…）
    下一步：py -3.11 release.py build --out <DIR> --napp "<WS>\work\cim-platform-1.0.0.napp" --blobs "<WS>\work\blobstore"

real	1m2.422s
user	0m0.000s
sys	0m0.046s

1-2. 簽章（發行者身分，production 必要）

> py -3.11 release.py sign work\cim-platform-1.0.0.napp --key keys\fab-team.private.json
OK: cim-platform-1.0.0.napp 已簽章（key_id=fab-team, algorithm=ed25519）

1-3. 組 release 資料夾（實測 9.8 秒）

--release-id 已存在會被拒絕——release 一律全新目錄，不就地增補。

> py -3.11 release.py build --out releases --napp work\cim-platform-1.0.0.napp --blobs work\blobstore --channel internal --release-id internal-20260719
OK: release 已建立 <WS>\releases\internal-20260719
    channel=internal  artifacts=1  payload=3 files
    出貨前最後一步：py -3.11 release.py verify "<WS>\releases\internal-20260719"

real	0m9.799s
user	0m0.016s
sys	0m0.062s

產出的 release 長這樣（offline-channel\ 就是 User 機器的 update source）：

releases\internal-20260719\ ├─ offline-channel\ │ ├─ channel.json ← 通道索引（Agent 讀這個） │ ├─ artifacts\cim-platform-1.0.0.napp │ └─ blobs\sha256\5d250bed… ← Tauri 殼（內容定址，跨版本共用） ├─ release-manifest.json / SBOM.json ├─ RELEASE-REPORT.md ← 人讀報告（含離線機三步驟） └─ checksums.sha256 ← 覆蓋除自身外的每一個檔案

1-4. 出貨前最後一問：可以出貨嗎？（實測 6.2 秒掃完 460MB）

> py -3.11 release.py verify releases\internal-20260719
OK: release 完整、與 manifest 一致，可出貨。

real	0m6.196s
user	0m0.015s
sys	0m0.031s

2. 晉升 production（強制驗章）

promote = 同一批 bytes 換通道全程重驗：production 通道拒絕未簽章或 驗不過章的 artifact，manifest 記錄 promoted_from 供追溯。（實測 15 秒）

> py -3.11 release.py promote releases\internal-20260719 --to-channel production --out releases --release-id production-20260719 --trust-store keys\trusted_publishers.json
OK: 已晉升為 production → <WS>\releases\production-20260719
    來源：internal-20260719

real	0m15.005s
user	0m0.015s
sys	0m0.031s

3. 搬到 User 機器：驗證 → 安裝 → 啟動

  1. 搬運：整個 production-20260719\ 複製到目標機（USB 可；別放 OneDrive 同步夾）。
  2. 先驗再裝：release.py verify <資料夾>——半份複製、被改過都在這裡被抓（見 §5）。

4. 之後的每一次更新（與回滾）

發 1.0.1 = 重複 §1–§2 的四步 + promote，然後 User 機器再跑一次 update。實測重點： 殼沒變 → blob 重用、零重複傳輸；user data（logs、tools.sqlite、選擇紀錄）一個 byte 都不動：

> rem User 機器:指向新的 production release,再跑一次 update
結果      : UPDATED
啟用版本  : 1.0.1
blob 下載 : 0/重用 1(殼沒變,零重複傳輸)

> rem 換版後,user data 一個 byte 都沒動:
data
engine.log
host.log
selected_paths.json
切換語意：update 只是把新版本裝好並指過去——正在跑的實例繼續用舊版直到重啟 （冷啟動切換）。回滾：新版啟動壞掉時 Agent 自動退回 last-known-good 並把壞版記入 failed_versions（不會反覆重試）；要手動退版用 agent.rollback()。 舊版本槽與舊殼由 Agent 的 GC 依「跨 App keep-set」回收，刪不掉的（防毒鎖住）記入 deferred 下輪再清，永不謊報已回收。

5. 出事的時候：搬運損壞／被動手腳

USB 搬到一半、或有人改了 release 裡任何一個 byte——verify 都會點名（本例 故意在 SBOM.json 加一個字元）：

> rem 模擬:USB 搬運途中 SBOM.json 被改了一個 byte
> py -3.11 release.py verify releases\tampered-demo
[FAIL] 雜湊不符：SBOM.json
共 1 個問題；此目錄**不可**出貨。
exit code: 1

更深一層：能寫入 update source 的攻擊者若「改 payload 後重生 files.json」讓逐檔雜湊 自洽，發行者簽章會把它擋在安裝前（這正是 §0 簽章存在的理由；有自動化測試釘住）。

6. 常見錯誤速查

症狀原因解法
[FAIL] 輸出目錄已存在重用了 release-id換新 id（如加 -r2）；release 不就地增補
production channel 拒絕未簽章.napp 沒簽或 verify 沒帶 trust store先 sign；verify/promote 帶 --trust-store
blob …缺失；被 cim-platform 引用--blobs 指錯或 blob 被搬走照訊息把該 sha256 檔放回 blobstore 再重跑
找不到 Tauri 殼（pack 時）prebuilt\cim-light.exe 不在非 WDAC 機器跑 build-shell.bat 後複製就位
User 機器 簽章的 key_id 不在信任清單新金鑰沒發布到裝置更新該機 trusted_publishers.json
啟動說 還沒有啟用中的平台版本裝置從未 update 過先跑 §3 的安裝那行
