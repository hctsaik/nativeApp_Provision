Runbook B — 打包成可交付資料夾

給 AI 照著執行。目標:把 CV Viewer + Tauri 殼 + 可攜 Python 合成一個資料夾, 拷給產線的人,雙擊就跑,不需要安裝 Python、Streamlit、Node 或 Rust。
工具是 native_Provision。基準版本 affe211(全套測試綠燈、交付鏈 E2E 12/12)。

先決定:你要哪一種佈局?兩種都能交付,差別在「第二個 App 進來的時候」。

fat(單一資料夾)Store(版本化 + 共用)
大小約 600 MB(每個 App 各自帶一份 runtime)共用 runtime 約 500 MB + 每個版本槽只有 10–100 MB
第二個 App再吃一次 600 MB+0 MB runtime(同一份 lock 就重用)
更新整包重發只發版本槽;原子更新 + 自動回滾
相依要求requirements.txt 或 pyproject 就行必須是完全釘死的 lock
什麼時候選它只有一個 App、只給幾台機器、先試試看要長期維護、要發新版、一台機器要放多個 App

CV Viewer 有 requirements.lock.txt(53 行,完全釘死)已驗證,兩種都能走。 這份文件兩條路都寫。

這件事跟 Runbook A 是「兩條獨立的路」,不要搞混。

  • Runbook A(開發):把 CV Viewer 註冊成 CIM 平台的一個工具 → 跑在完整的平台裡(engine + portal + 所有其他工具)。
  • Runbook B(交付):把 CV Viewer 單獨包成一個桌面 App → 殼裡只有它一個。 它不需要 Runbook A 的 plugin.yaml / runner,也不會把 CIM 平台的其他工具帶過去。

兩者共用同一顆 cim-light.exe,但走的是不同的 engine:交付包裡跑的是 native_Provision 自己的 engine_shim.py,只服務這一個 App。

⚠ 選用相依「torch」沒有宣告,但只在 5_PG_Develop/dinodiff.py:244(函式內延遲 import)用到,

這個警告是對的,而且很重要:torch 在 dinodiff.py 的函式裡面才 import, 所以 App 啟動不會炸 —— 閘門因此只警告、不擋建置(擋了就會誤殺一個能跑的 App)。 但那條「用 DINOv2 做比對」的功能路徑會在使用者點下去的時候才失敗。

你要做的決定:那個功能要不要交付?要 → 把 torch(和 torchvision)加進 lock 再建置。 不要 → 就這樣出,但你已經知道它會缺。

檢查專案會跳出「這個專案裡有一些大東西」—— 這是對的,不是 bug。 已驗證

⚠ 專案裡的大檔:models/dinov2_vits14.pth(84 MB)——確定要交付嗎?
· 已自動排除:wheels/ 124 MB、.git/ 11 MB、__pycache__/ 3 MB(共 138 MB,不會進交付包)

那 84 MB 是 DINOv2 的權重,App 離線執行真的需要它 → 按「確定」帶進去。
而 wheels/、.git/ 是自動排掉的,只是資訊,不需要你做任何決定。

WebView2:這是交付包裡「唯一」需要事先裝在目標機上的東西。

應用視窗是 WebView2 畫出來的。Win11 大多內建;但無網工廠機如果沒有, App 會開不起來(start.bat 會先偵測並 exit 5),而且當場沒網路可以裝。

到 go.microsoft.com/fwlink/?LinkId=2124701 下載 Evergreen Standalone Installer,在「進階設定 → WebView2 離線安裝檔」指定它。 它會被放進交付包的 prereq\,tools\安裝WebView2.bat 就能離線裝(不需管理員權限)。

沒填的話,建置不會假裝成功 —— 完成對話框會警告你這件事。 而且已經建好的包不必重建:把安裝檔複製進 <交付包>\prereq\ 就行。

路線二:程式碼(給 AI 自動化用)

fat 佈局 —— 一個資料夾,雙擊就跑

# build_cv_viewer_fat.py
import sys
sys.path.insert(0, r"C:\code\claude\native_Provision\src")

from pathlib import Path
from provision_builder.streamlit_desktop import (
    BuildRequest, build, validate_request, warnings_for,
    find_entrypoint, find_runtime, find_shell, suggest_name,
)
from provision_builder.streamlit_desktop.builder import scan_project

PROJECT = Path(r"C:\code\claude\CV_Viewer")

request = BuildRequest(
    project_dir=PROJECT,
    entrypoint=find_entrypoint(PROJECT).value,          # → 5_PG_Develop\app.py
    display_name=suggest_name(PROJECT),                 # → "CV Viewer"(保留大小寫)
    output_dir=Path(r"C:\code\claude\native_Provision\dist\streamlit-apps"),
    shell_exe=find_shell().value,                       # → nativeApp 的 cim-light.exe
    runtime_template=find_runtime().value,              # → .runtime-cache\python311
    preferred_port=0,                                   # 0 = 每次啟動隨機挑一個沒被占用的埠
    requirements=PROJECT / "requirements.lock.txt",
    # webview2_installer=Path(r"C:\...\MicrosoftEdgeWebView2RuntimeInstallerX64.exe"),
)

# 1) 閘門:0 秒就給答案,不要等 6 分鐘的 pip 裝完才炸
errors = validate_request(request)
if errors:
    raise SystemExit("檢查未通過:\n  " + "\n  ".join(errors))
for w in warnings_for(request):
    print("⚠", w)

# 2) 大檔在「開工前」就知道
scan = scan_project(request)
print(f"專案 {scan.application_mb:.0f} MB")
for w in scan.warnings:  print("⚠", w)     # 需要你決定的(84MB 權重檔)
for n in scan.notes:     print("·", n)     # 純資訊(已自動排除的 138MB)

# 3) 建置(progress 會逐行吐出來,包含 pip 的輸出)
result = build(request, progress=lambda line: print("   ", line))
if not result.ok:
    raise SystemExit(f"建置失敗:{result.errors}\n完整記錄:{result.log_path}")

print(f"\n交付資料夾:{result.package_dir}")
print(f"大小 {result.size_mb:.0f} MB · {result.duration_seconds:.0f} 秒")
for w in result.warnings:
    print("⚠", w)          # ← 沒附 WebView2 安裝檔的話,警告會出現在這裡

Store 佈局 —— 版本化、共用 runtime、可回滾

# build_cv_viewer_store.py（承上,request 相同）
from provision_builder.streamlit_desktop import build_into_store, export_full_tree

STORE = Path(r"C:\code\claude\native_Provision\dist\cv-store")

result = build_into_store(request, STORE, version="v1.0.0",
                          progress=lambda line: print("   ", line))
if not result.ok:
    raise SystemExit(result.errors)

print(f"版本 {result.version}:{result.version_mb:.0f} MB")
print(f"本次磁碟新增 {result.added_mb:.0f} MB(runtime 重用 = {result.runtime_reused})")

# 匯出「完整交付」給新機器 —— 整棵可執行的樹
export = export_full_tree(STORE, Path(r"D:\deliver"), app_id=result.app_id,
                          version="v1.0.0",
                          progress=lambda line: print("   ", line))
print(f"交付物:{export.out_dir}({export.total_mb:.0f} MB)")
print(f"User 要雙擊的是:{export.entry_hint()}")
for w in export.warnings:
    print("⚠", w)

AI 最容易犯的一個錯:匯出時忘了傳 version=。

不傳的話,export_full_tree 會去交付 state.current —— 而在建置機上,剛建好的新版是設成 pending(建置機自己不會去啟動它), 所以 current 是舊的那一版。 結果就是:你選了 v1.1.0、走一趟工廠、裝上去的還是 v1.0.0。

用 newest_version(STORE, app_id) 拿「最新且完整」的版本,或明確指定。

發新版給已經交付出去的機器

這是 Store 佈局存在的理由。不要重發 600 MB。

# 建置機:建 v1.1.0(同一份 lock → runtime 重用 → 只多一個版本槽)
build_into_store(request, STORE, version="v1.1.0", ...)

# 匯出「更新包」(只有這一版,十幾 MB)

upd = export_update(STORE, app_id, "v1.1.0", Path(r"D:\update"), include_runtime=False)

然後在目標機上(交付樹自己就帶著 bootstrap):

rem 套用更新:複製 → 逐檔驗 files.json → 驗過才寫 .complete → 設為下次啟動套用
python bootstrap\bootstrap.py --install D:\update\app-cv-viewer

rem 或者用選單:tools\admin.bat → [3] 套用已複製進來的更新包

目標機能自己做的事 已驗證(bootstrap.py --help)

--status目前版本 / 上一版 / 最後可用 / 待套用 / 更新來源 / 失敗紀錄(一頁,可以念給電話另一頭聽)
--install <包>安裝更新包。驗證不過就完全不動系統
--clear-pending裝好了但改變主意 → 取消(版本留在磁碟上,之後還能再套用)
--rollback立刻退回上一個能用的版本。沒東西可退就明講並失敗,不會假裝成功
--rollback-to <版本>退到指定版本
--clear-failed <版本>修好之後,讓它可以再試一次
--set-update-source <路徑>指向網路芳鄰/USB,之後自動更新

安全網:壞版上不了線,上了線也退得回來

  • 建置端:import 閉包掃描(含 Streamlit 多頁 App 的 pages/)—— App 會 import 的東西,runtime 裡沒有,直接讓建置失敗,不會等到產線才炸。
  • 啟動端:launcher 在 spawn 任何東西之前,先用 App 自己的 runtime 做一次 import 預檢。 /_stcore/health 是伺服器回的 —— App 死在 import cv2 照樣回 200,所以它不能當唯一的健康訊號。
  • 晉升為 last-known-good 的條件是:正常結束 + healthy marker 還在。 不是「marker 一出現就蓋章」—— App 腳本要等 User 按 Start 才跑,可能是幾分鐘之後的事。
  • exit code 分流:3=App 壞了、4=版本樹壞了、5=這台機器壞了(缺 WebView2/防毒)。 殼是所有版本共用的 —— 把環境失敗算在版本頭上,會標死一個好版本,再「回滾」到一樣開不起來的版本。

GC 的 exit code 是分得開的 已驗證 —— 「乾淨的樹」跟「回收失敗」不是同一回事:

0有東西可回收 / 全部刪掉了
2部分回收(有些檔案被 App 佔用著)
3試了,一個位元組都沒回收 → 有問題
4拿不到鎖,根本沒掃
6沒有可回收的項目(乾淨的樹) —— 這不是失敗

它報實際刪掉的量,不報預測值。刪不掉的會列出來並說原因(檔案使用中 → 關掉 App 再跑一次)。

已知限制(不要花時間去「修」它們)

限制為什麼
User 雙擊之後,還要在「工作流程」下拉按一次 Start portal 烤在預建 exe 裡。要自動啟動就得重編殼,而本機 WDAC 擋重編(os error 4551)。要在沒有 WDAC 的機器上重編殼才能改掉。
fat 包約 600 MB Streamlit 生態系的地板(pandas / pyarrow / numpy…)。Store 佈局就是為了讓第二個 App 不用再吃一次。
Store 佈局要求完全釘死的 lock 「共用同一份 runtime」的前提是「兩邊的相依完全一樣」。指紋是從釘死的 pin 算出來的。
WebView2 要事先裝在目標機上 它是系統元件,不是我們能塞進資料夾的東西。所以我們把離線安裝檔塞進去。
